5 /*
6  The application programmer can append application specific data to each
7  library viewport by defining GEXTMODE in gdispcfg.h
8  The GVPAPP data structure defined below is then added to each viewport.
9  When GEXTMODE is defined the GVPAPP structure must contain at least one
10  data member.
11 
12  You can get a pointer to the application specific data when you have the
13  viewport number or get a the viewport number if you have a (valid) pointer
14  to the GVPAPP structure:
15  PGVPAPP ggetapp( void ) Get pointer to GVPAPP for current viewport
16  PGVPAPP ggetapp_vp( SGUCHAR vp ) Get pointer to GVPAPP for specified viewport
17  SGUCHAR ggetvpapp( PGVPAPP app ) Get viewport number for PGVPAPP
18 
19  The grapich library itself does not use any data members of the GVPAPP
20  structure. It only refer to the structure via a pointer. You may therefore
21  define the structure content as you like to fit your application.
22 
23  Note:
24  The ginit() function does not modify or use any of the GVPAPP data members.
25  The GVPAPP structures can therefore be initiated for the first time
26  either before or after the initial ginit() call.
27 
28  The initialization of all viewport GVPAPP's can be done like this:
29 
30  #include <gdisp.h>
31  //...
32  void my_vpapp_init(void)
33  {
34  SGUCHAR i;
35  for (i = 0; i < GNUMVP; i++)
36  {
37  PGVPAPP app;
38  app = ggetappvp(i); // Get pointer to GVPAPP stucture for viewport number
39  // Do first-time initialization of the GVPAPP structure members
40  // via the app pointer
41  }
42  }
43 
44 
45  Note:
46  As data members defined here in GVPAPP are added to all viewports then care
47  should be taken not to overuse the feature.
48  For instance just adding a generic pointer to application specific data
49  structures may be the right way to optimize RAM memory consumption in some
50  applications.
51  In other system using a union of application specific data structures may be
52  the best solution to optimize the RAM memory consumption.
53 */
